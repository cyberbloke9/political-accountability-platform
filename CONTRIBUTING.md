# Contributing to Political Accountability Platform

First off, thank you for considering contributing to the Political Accountability Platform! It's people like you who make this platform a powerful tool for democratic accountability.

### 1. Report Bugs üêõ
Found a bug? Help us fix it!

- Check if the bug has already been reported in [Issues](https://github.com/cyberbloke9/political-accountability-platform/issues)
- If not, create a new issue with:
  - Clear, descriptive title
  - Steps to reproduce the bug
  - Expected vs actual behavior

### 2. Suggest Features üí°
Have an idea to improve the platform?

- Check existing [feature requests](https://github.com/cyberbloke9/political-accountability-platform/issues?q=is%3Aissue+label%3Aenhancement)
- Open a new issue with the `enhancement` label
- Describe the feature and its benefits
- Include mockups or examples if applicable

### 3. Submit Code üîß

#### Getting Started

1. **Fork the repository**
   ```bash
   gh repo fork cyberbloke9/political-accountability-platform
   ```

2. **Clone your fork**
   ```bash
   git clone https://github.com/YOUR_USERNAME/political-accountability-platform.git
   cd political-accountability-platform
   ```

3. **Create a branch**
   ```bash
   git checkout -b feature/your-feature-name
   # or
   git checkout -b fix/your-bug-fix
   ```

4. **Set up the development environment**
   - Follow the setup instructions in [README.md](./README.md)
   - Make sure all tests pass: `npm test`

#### Code Style

We use consistent code style across the project:

- **TypeScript**: Strict mode enabled
- **Formatting**: Prettier with 2-space indentation
- **Linting**: ESLint with Next.js recommended rules
- **Naming**: 
  - Components: PascalCase (e.g., `PromiseCard.tsx`)
  - Files: kebab-case (e.g., `use-auth.ts`)
  - Variables/Functions: camelCase

#### Making Changes

1. **Write clean, readable code**
   - Add comments for complex logic
   - Use meaningful variable names
   - Keep functions small and focused

2. **Test your changes**
   - Ensure the app builds: `npm run build`
   - Test manually in development: `npm run dev`
   - Write tests for new features (coming soon)

3. **Commit your changes**
   ```bash
   git add .
   git commit -m "feat: add promise filtering by constituency"
   ```

   Use conventional commits:
   - `feat:` - New features
   - `fix:` - Bug fixes
   - `docs:` - Documentation changes
   - `style:` - Code style changes (formatting, etc.)
   - `refactor:` - Code refactoring
   - `test:` - Adding tests
   - `chore:` - Maintenance tasks

4. **Push to your fork**
   ```bash
   git push origin feature/your-feature-name
   ```

5. **Open a Pull Request**
   - Go to the original repository
   - Click "New Pull Request"
   - Select your fork and branch
   - Fill in the PR template:
     - Description of changes
     - Related issue number (if applicable)
     - Screenshots (for UI changes)
     - Testing done

## üö´ What NOT to Contribute

To maintain quality and focus:

- **Don't** submit AI-generated code without review and testing
- **Don't** add unnecessary dependencies
- **Don't** make breaking changes without discussion
- **Don't** include personal credentials or API keys
- **Don't** submit incomplete work (use draft PRs instead)

## üîç Code Review Process

1. **Automated Checks**: CI/CD runs linting, type checking, and build
2. **Manual Review**: Maintainers review code quality and design
3. **Testing**: Changes are tested in a staging environment
4. **Approval**: At least one maintainer approval required
5. **Merge**: Squash and merge to main branch

Reviews typically take 2-5 days. Be patient and responsive to feedback!

## üìú Code of Conduct

### Our Pledge

We are committed to making participation in this project a harassment-free experience for everyone, regardless of:
- Age, body size, disability
- Ethnicity, gender identity and expression
- Level of experience, nationality
- Personal appearance, race, religion
- Sexual identity and orientation

### Our Standards

**Positive behavior includes:**
- Using welcoming and inclusive language
- Being respectful of differing viewpoints
- Gracefully accepting constructive criticism
- Focusing on what's best for the community
- Showing empathy toward others

**Unacceptable behavior includes:**
- Harassment, trolling, or derogatory comments
- Publishing others' private information
- Political attacks or inflammatory discussions

### Enforcement

Violations may result in:
1. **Warning**: Private message explaining the violation
2. **Temporary Ban**: Suspension from the project for a period
3. **Permanent Ban**: Removal from the project indefinitely

Report violations to [papsupport@gmail.com](mailto:papsupport@gmail.com)
